## Как проверять статус ботов (Self-Check)
ВАЖНО: Процессы ботов НЕ содержат имя бота в командной строке! Все выглядят как `Python telegram_bot.py` или `Python bot.py`.

**Правильный способ проверки:**
1. **Через PID файлы** (самый надёжный):
   - Василий: `cat ~/logs/vasily-bot.pid` → проверить `ps -p <PID>`
   - Маша: `cat ~/logs/masha-bot.pid` → проверить `ps -p <PID>`
   - Костя: `cat ~/logs/kostya-bot.pid` → проверить `ps -p <PID>`
   - Beast: `cat ~/logs/beast-bot.pid` → проверить `ps -p <PID>`
2. **Nexus/ClaudeClaw**: `launchctl list | grep claudeclaw` или `ps aux | grep claudeclaw | grep -v grep`
3. **Через LaunchAgent**: `launchctl list | grep vladimir`
4. **НЕ ДЕЛАЙ** `ps aux | grep vasily` — это НЕ работает, в процессе нет имени бота!

## Универсальные правила работы с кодом (Karpathy Skills)

Применяются поверх проектных правил выше. **Компромисс:** эти правила всегда смещают баланс в сторону осторожности, а не скорости. Для тривиальных задач используй здравый смысл.

### 1. Думай прежде чем писать код

**Не угадывай. Не скрывай неопределённость. Озвучивай компромиссы.**

Перед реализацией:
- Явно формулируй допущения. Если не уверен — спроси.
- Если возможны несколько интерпретаций — предложи их, не выбирай молча.
- Если есть более простой подход — скажи об этом.
- Если что-то непонятно — остановись, назови что именно, спроси.

### 2. Простота прежде всего

**Минимум кода для решения задачи. Ничего спекулятивного.**

- Никаких фич сверх запрошенного.
- Никаких абстракций для одноразового кода.
- Никакой «гибкости» или «настраиваемости», которую не просили.
- Никакой обработки ошибок для невозможных сценариев.
- Если написал 200 строк, а можно было 50 — перепиши.

Вопрос к себе: «Сказал бы старший инженер, что это переусложнено?» Если да — упростить.

### 3. Хирургические изменения

**Трогай только то, что необходимо. Убирай только свой мусор.**

При редактировании существующего кода:
- Не «улучшай» соседний код, комментарии или форматирование.
- Не рефактори то, что не сломано.
- Соблюдай существующий стиль, даже если ты бы сделал иначе.
- Если видишь мёртвый код не по теме — упомяни, но не удаляй.

Свои изменения:
- Удаляй импорты/переменные/функции, которые стали неиспользуемыми ТОЛЬКО из-за твоих изменений.
- Не трогай уже существующий мёртвый код без явного запроса.

Тест: каждая изменённая строка должна прямо следовать из запроса пользователя.

### 4. Выполнение, ориентированное на цель

**Определяй критерии успеха. Итерируй до проверки.**

Преобразуй задачи в проверяемые цели:
- «Добавь валидацию» → «Напиши тесты для невалидных входных данных, затем заставь их пройти»
- «Починить баг» → «Напиши тест, воспроизводящий баг, затем заставь его пройти»
- «Отрефактори X» → «Убедись, что тесты проходят до и после»

Для многошаговых задач — коротко опиши план:
```
1. [Шаг] → проверка: [критерий]
2. [Шаг] → проверка: [критерий]
```
